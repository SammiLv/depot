# 部门内部管理网站数据库表设计

## 1. 设计目标

本数据库设计服务于部门内部管理网站 MVP 阶段，优先满足以下目标：

```text
1. 支持产品部 15 人试点。
2. 支持后续扩展至公司内部 150 人以内。
3. 支持 SQLite 本机开发和试部署。
4. 通过 Prisma 保留迁移 PostgreSQL 的能力。
5. 表结构清晰，便于后续功能扩展。
6. 不过度设计复杂企业级模型。
```

## 2. 数据库技术方案

MVP 阶段：

```text
SQLite + Prisma
```

后续正式推广：

```text
PostgreSQL + Prisma
```

建议所有数据库访问都通过 Prisma 完成，避免在代码里大量写原生 SQLite SQL。

## 3. 通用设计约定

### 3.1 主键

所有核心业务表建议使用字符串主键：

```text
id String @id @default(cuid())
```

原因：

```text
SQLite 和 PostgreSQL 都支持
便于后续迁移
避免自增 ID 在导入导出时冲突
```

### 3.2 时间字段

大部分表建议包含：

```text
createdAt
updatedAt
```

部分业务表额外包含：

```text
submittedAt
approvedAt
reviewedAt
completedAt
```

### 3.3 删除策略

MVP 阶段建议优先使用软删除：

```text
deletedAt DateTime?
```

特别是：

```text
用户
指标
KPI
季度工作
人才记录
```

避免误删历史数据。

### 3.4 状态字段

状态字段建议使用枚举：

```text
roleType
approvalStatus
workStatus
riskStatus
kpiStatus
reviewGrade
rewardType
```

Prisma 中可以定义 enum。

## 7. 年度业绩指标管理表

## 7.1 年度指标业务结构

当前实现下，年度指标不是单一目标值，而是“方案 + 指标项 + 元指标 + 季度目标”的分层结构。

示例：产品部 2026 年度业绩指标：

```text
主营业务业绩分值：目标 267，权重 50%
创新 ToB 营收：目标 500w，权重 20%
创新 ToC 用户注册数：目标 35000 人，权重 30%
```

最终完成度：

```text
(主营业务实际分值 / 267) * 50%
+ (ToB 实际营收 / 500w) * 20%
+ (ToC 实际注册数 / 35000) * 30%
```

当前代码对应的年度指标相关模型为：

```text
AnnualGoalPlan            年度指标方案
AnnualGoalMetric          方案内的年度指标项
AnnualGoalMetricSource    指标项下的元指标
AnnualGoalQuarterTarget   指标项或元指标的季度拆解目标
AnnualGoalProgress        进度更新记录模型（当前主要为预留）
AnnualGoalRevisionLog     修订日志模型（当前主要为预留）
```

当前层级关系：

```text
方案（Plan）
  -> 指标项（Metric）
    -> 元指标（Source，可选）
      -> 季度目标（QuarterTarget）
```

说明：
- 部门方案可以直接维护指标项，并继续拆解元指标。
- 小组方案当前不直接维护元指标主数据，而是从本部门方案中选择指标项或元指标承接。
- 权重、元指标目标总额、季度目标总额当前都校验“不得超额”，但不强制“必须补满”。

### 7.1.1 AnnualGoalPlan 年度指标方案表

用途：保存某一年、某个部门或小组的一套年度指标方案。当前已支持部门方案、小组方案、历史归档，以及版本/修订字段预留。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| id | String | 是 | 主键 |
| year | Int | 是 | 年度 |
| name | String | 是 | 方案名称 |
| description | String? | 否 | 方案说明 |
| ownerType | AnnualGoalOwnerType | 是 | 归属类型：部门/小组 |
| departmentId | String? | 否 | 所属部门 |
| teamId | String? | 否 | 所属小组，部门方案为空 |
| parentPlanId | String? | 否 | 上级方案 ID，当前模型已预留 |
| version | Int | 是 | 版本号，当前默认 1 |
| isActive | Boolean | 是 | 是否当前启用 |
| approvalStatus | ApprovalStatus | 是 | 审批状态字段，当前主要为预留 |
| effectiveFrom | DateTime? | 否 | 生效开始时间 |
| effectiveTo | DateTime? | 否 | 生效结束时间 |
| approvedAt | DateTime? | 否 | 审批通过时间 |
| revisionReason | String? | 否 | 修订原因 |
| revisedFromPlanId | String? | 否 | 来源旧版本方案 ID |
| createdById | String | 是 | 创建人 |
| createdAt | DateTime | 是 | 创建时间 |
| updatedAt | DateTime | 是 | 更新时间 |
| deletedAt | DateTime? | 否 | 软删除时间，用于归档 |

ownerType：

```text
DEPARTMENT  部门年度指标方案
TEAM        小组年度指标方案
```

当前业务约束：
- 方案年份必填，且必须在 2000~2100 之间
- 方案名称必填
- 部门方案必须有所属部门
- 小组方案必须有所属小组
- 当前只有管理员、部门主管可以维护方案
- 部门主管只能维护本部门范围内的方案

- 删除元指标时，已承接该元指标的小组指标会同步删除

### 7.1.3 AnnualGoalMetric 年度指标项表

用途：保存年度指标方案中的具体指标项。当前既支持部门方案直接维护，也支持小组方案引用部门指标或部门元指标形成映射项。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| id | String | 是 | 主键 |
| planId | String | 是 | 所属年度指标方案 |
| sourceMetricId | String? | 否 | 若为小组承接元指标，则指向来源元指标 |
| metricCode | String | 是 | 指标编码 |
| name | String | 是 | 指标名称 |
| description | String? | 否 | 指标说明 |
| targetValue | Float | 是 | 年度目标值 |
| currentValue | Float | 是 | 当前累计完成值 |
| unit | String | 是 | 单位，如 分、元、人 |
| weight | Float | 是 | 权重，当前按百分比存储，如 50 |
| calculationType | AnnualMetricCalculationType | 是 | 计算方式 |
| riskStatus | RiskStatus | 是 | 风险状态 |
| sortOrder | Int | 是 | 排序 |
| adjustedAt | DateTime? | 否 | 最近调整时间 |
| progressUpdatedAt | DateTime? | 否 | 最近进度更新时间 |
| createdAt | DateTime | 是 | 创建时间 |
| updatedAt | DateTime | 是 | 更新时间 |
| deletedAt | DateTime? | 否 | 软删除时间 |

calculationType：

```text
RATIO        按 currentValue / targetValue 展示
BOOLEAN      布尔型
MANUAL_SCORE 手工评分型
```

当前业务约束：
- 指标项权重必须为非负数
- 同一方案下所有未删除指标项权重合计不得超过 100%
- 当前允许权重未补满，不要求必须等于 100%
- 小组方案创建指标项时，必须二选一：选择部门指标项，或选择部门元指标
- 小组方案不能引用其他部门的指标或元指标
- 小组方案承接后，名称、目标值、单位、计算方式、风险状态等来自上游指标或元指标
- 小组方案后续主要调整权重和排序

## 17. 设计结论

本数据库设计采用“核心业务闭环优先”的思路：

```text
组织权限
审批流
年度指标
季度工作
KPI
人才发展
通知待办
```

表结构既能支撑当前产品部试点，又保留后续迁移 PostgreSQL、接入钉钉、扩展多部门和新增业务模块的空间。
