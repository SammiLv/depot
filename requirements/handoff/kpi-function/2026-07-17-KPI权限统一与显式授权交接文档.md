已在：

本轮变化的核心不是“再加一个 if”，而是把原本偏向单一 role-scope 的逻辑，收口成真实的覆盖结果计算。

  - `orgNodeIds`

- `buildKpiWhereByPermission(...)` 改为按 coverage 组装 where

### 2.5 显式授权服务端已支持批量创建与节点归并

已在：

- `src/server/organization/actions.ts`

完成显式授权创建逻辑升级。

这一轮已经支持：

- 多用户
- 多能力项
- 多节点
- 提交按 **用户 × 能力项 × 节点** 全组合展开

已落地的服务端关键点：

1. 通过 `FormData.getAll(...)` 解析数组字段
2. 统一做去重
3. 校验目标用户是否合法、是否在当前可管理范围内
4. 校验目标节点是否合法、是否在当前可管理范围内
5. 对节点做归并：
   - 若同时选中某部门和其下级小组
   - 优先保留更宽的部门级节点
   - 避免生成冗余 grant
6. 在事务中先删同组合旧记录，再批量写入新 grant

这意味着“批量授权”已经不是前端假多选，而是后端真实支持批量展开落库。

## 7. 风险点与注意事项

### 7.1 显式授权列表仍按原子 grant 行展示

这是当前有意保留的策略，不是遗漏。

原因：

- 当前删除逻辑按单条 grant 删除最安全
- 如果在列表层先把多条 grant 聚合成一行，会带来“删一行到底删哪些底层记录”的歧义

所以当前的取舍是：

- 创建流程支持批量
- 列表展示仍保持原子
- 删除仍保持精确

如果后续要改成更友好的聚合展示，建议同时重新定义删除语义。

### 7.2 旧的 `ALL` 显式授权记录仍可能存在

虽然当前最新弹窗已经不再创建“所有部门”显式授权，但历史数据里若已经存在：

- `scopeType = ALL`
- `orgNodeId = null`

前端列表仍会显示为：

- 范围：`全部`
- 节点：`全部`

这是兼容旧数据的有意保留，不是今天这轮漏删。

### 7.3 当前工作区尚未提交

今天这轮代码目前都还在工作区里，没有形成新的提交。

因此如果后续要切分提交或合并到本地 `main`，需要注意：

- 当前改动面较大
- 涉及 schema / migration / resolver / UI / query 多层联动
- 提交前最好先再做一次页面验证，确认简化后的显式授权弹窗无回退问题
