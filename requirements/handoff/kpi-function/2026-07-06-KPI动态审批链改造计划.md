# KPI 动态审批链改造计划

**日期**：2026-07-06  
**分支**：depot-KPI  
**目标**：将 KPI 审批流从固定枚举顺序（SELF → LEADER → MANAGER → FINAL）改为按单据生成审批人快照 + 逐节点流转，解决"组长自己的 KPI 还需走一遍组长评"的问题。

---

## 核心思路

KPI 初始化时，按被评人所在组织节点向上解析一次审批链，生成快照记录。后续流转时，按快照里的节点顺序推进，而不是走固定枚举顺序。被评人自己出现在链路中的节点自动跳过。

---

## 第一步：新增 Schema — `PersonalKpiApprovalStep`

**文件**：`db/prisma/schema.prisma`

新增一张表，存储单据级别的审批链快照：

```prisma
model PersonalKpiApprovalStep {
  id            String    @id @default(cuid())
  personalKpiId String
  stepOrder     Int       // 1=组长评, 2=主管评, 3=终审
  stageKey      String    // "LEADER" | "MANAGER" | "FINAL"
  approverId    String    // 快照时确定的审批人 userId
  status        String    @default("PENDING") // PENDING | COMPLETED | SKIPPED | REJECTED
  completedAt   DateTime?
  createdAt     DateTime  @default(now())

  @@index([personalKpiId, stepOrder])
  @@index([approverId, status])
}
```

同步写迁移文件并执行 `npx prisma migrate deploy --config "db/prisma.config.ts"`。

---

## 第二步：实现链路解析逻辑

**新文件**：`src/server/kpi/approval-chain.ts`

实现 `resolveApprovalChain(userId, orgNodeId)` 函数，返回有序的 `ApprovalStep[]`：

1. **组长评**：找 `roleType = TEAM_LEADER` 且 `orgNodeId = 被评人的 orgNodeId` 的用户。若该用户就是被评人本人 → 跳过此步。
2. **主管评**：找 `roleType = DEPARTMENT_MANAGER` 且所在部门与被评人同部门的用户。若与上一步同一人或就是被评人 → 跳过。
3. **终审**：找 `roleType = ADMIN` 的第一个用户（后续可配置）。若与前两步重复 → 跳过。

跳过逻辑只过滤"同一人"，不过滤"链路为空"的情况——如果整个链路都是自己，链路为空，KPI 自评提交后直接流转到 `COMPLETED`。

## 第三步：初始化时写入链路快照

**文件**：`src/server/kpi/actions.ts`，改 `createPersonalKpiSnapshot()`

在创建 `PersonalKpi` 记录之后，紧跟调用 `resolveApprovalChain`，把结果写入 `PersonalKpiApprovalStep`：

```ts
// 在 createPersonalKpiSnapshot 末尾
const chain = await resolveApprovalChain(input.userId, input.orgNodeId);
if (chain.length > 0) {
  await tx.personalKpiApprovalStep.createMany({
    data: chain.map((step, i) => ({
      personalKpiId: createdKpi.id,
      stepOrder: i + 1,
      stageKey: step.stageKey,
      approverId: step.approverId,
      status: "PENDING",
    })),
  });
}
```

## 第四步：重写流转逻辑

**文件**：`src/server/kpi/actions.ts`，改 `getNextKpiStage()` / `getPreviousKpiStage()`

### 提交/通过（向后流转）

```
当前状态 PENDING_SELF_REVIEW → submit
  1. 查 PersonalKpiApprovalStep
     where personalKpiId = X AND status = "PENDING"
     ORDER BY stepOrder ASC LIMIT 1
  2. 若有 → nextStatus 按 step.stageKey 映射：
       LEADER  → PENDING_LEADER_SCORE
       MANAGER → PENDING_MANAGER_SCORE
       FINAL   → PENDING_FINAL_REVIEW
  3. 若无 → nextStatus = COMPLETED
  4. 把当前完成的 step 标记为 COMPLETED
```

### 退回（向前流转）

```
任意审批阶段 → reject
  1. 把当前 step 标记为 REJECTED
  2. nextStatus = PENDING_SELF_REVIEW（统一退回自评，不逐级退回）
```

退回统一退到自评，简单可预期，不做逐级退回。

### stageKey → KpiStatus 映射表

| stageKey  | KpiStatus                |
|-----------|--------------------------|
| LEADER    | PENDING_LEADER_SCORE     |
| MANAGER   | PENDING_MANAGER_SCORE    |
| FINAL     | PENDING_FINAL_REVIEW     |

## 第五步：收紧评分权限校验

**文件**：`src/server/kpi/actions.ts`，改 `persistPersonalKpiScoring()`

当前的权限校验是 `buildKpiWhereByPermission(角色+能力)`，只校验角色。改造后需要额外校验：

```ts
// 在 LEADER / MANAGER / FINAL 阶段，额外检查当前人是否就是该步骤的 approverId
const currentStep = await tx.personalKpiApprovalStep.findFirst({
  where: { personalKpiId, status: "PENDING" },
  orderBy: { stepOrder: "asc" },
});
if (currentStep && currentStep.approverId !== currentUser.id) {
  throw new Error("你不是当前阶段的审批人");
}
```

SELF 阶段保持原有逻辑（`currentUser.id === personalKpi.userId`）。

**文件**：`src/server/kpi/kpi-query.ts`，改 `getPersonalKpiDetail()`

新增查询 `PersonalKpiApprovalStep`，返回每个步骤的审批人姓名，前端在流程进度条下展示"审批人：张三"。

## 第七步：存量数据兼容

**策略**：存量 KPI（已有 `status` 的记录）不做链路回填，继续用现有枚举流转。

fallback 判断逻辑：

```ts
const hasChain = await tx.personalKpiApprovalStep.count({
  where: { personalKpiId }
}) > 0;

// hasChain === false → 走旧的 getNextKpiStage() 枚举逻辑
// hasChain === true  → 走新的快照流转逻辑
```

新旧逻辑并存，新初始化的单据走新路径，存量单据走旧路径，不影响进行中的 KPI 流程。

## 约束说明

- **不动 `KpiStatus` 枚举**：前端 tab/筛选/进度条的 stageKey 映射关系不变。
- **不改模板/初始化入口**：只在快照写入时插入新逻辑。
- **不影响权限配置**：`OrgPermissionGrant` 表和 `resolvePermissionScope` 逻辑不变，新增的是单据级别的审批人匹配。
- **链路为空时的兜底**：若某人组内无组长、部门无主管、系统无 ADMIN，对应步骤自动跳过，不阻塞流程。
