# 权限作用域重构可执行清单

## 目标

本次改造的目标不是重做用户角色体系，而是在现有 `User.roleType` 不变的前提下，把权限配置从“按角色全局配置”改成“按作用域配置”：

- `SYSTEM`：系统默认权限模板
- `DEPARTMENT`：部门级权限覆盖

最终生效规则采用 **系统默认 + 部门覆盖**：

1. 先读取当前部门是否存在显式权限配置
2. 若存在，则使用部门配置
3. 若不存在，则回退到系统默认配置

---

## 一、数据模型改造

### 1. 扩展权限表作用域字段

目标表：

- `RoleMenuPermission`
- `RoleAnnualGoalPermission`

当前两张表都是全局模型：

- `roleType + permissionId`

需要改成带作用域的模型，建议新增：

- `scopeType`：枚举，建议 `SYSTEM | DEPARTMENT`
- `departmentOrgNodeId`：仅部门作用域使用，系统作用域为空

### 2. 调整唯一约束

唯一键从：

- `roleType + permissionId`

改成：

- `scopeType + departmentOrgNodeId + roleType + permissionId`

注意：

- `SYSTEM` 记录中 `departmentOrgNodeId = null`
- `DEPARTMENT` 记录中 `departmentOrgNodeId = 当前部门的 OrgNode.id`

### 3. 明确存储语义

一条记录不是“一个部门的整张权限表”，而是：

- 某个作用域
- 某个部门（若为部门作用域）
- 某个角色
- 某个权限项
- 的一条显式配置

需要额外确认当前表是否继续沿用“有记录=允许”的加法授权语义。

如果要完整支持：

- 系统默认允许 / 禁止
- 部门显式允许 / 显式禁止
- 无部门记录时继承系统

则需要评估是否新增 `allowed` 布尔值，否则当前“存在即允许”模型难以表达“部门显式关闭”。

---

## 二、迁移与种子数据

### 4. 编写 migration

迁移原则：

- 将当前全局 `RoleMenuPermission` 记录整体迁移到 `SYSTEM` 作用域
- 将当前全局 `RoleAnnualGoalPermission` 记录整体迁移到 `SYSTEM` 作用域
- 保证迁移后默认行为不变

### 5. 控制部门初始数据策略

不要给每个部门预复制一套系统权限矩阵。

原因：

- 会造成脏数据膨胀
- 后续系统默认变更无法自然传递
- 难以区分“继承”与“显式配置”

推荐策略：

- 系统层存默认权限
- 部门层只存显式覆盖记录
- 未配置则运行时回退到系统默认值

### 6. 更新 seed

需要同步调整 `db/prisma/seed.ts`：

- 初始化系统默认权限时，写入 `SYSTEM` 作用域
- 不再按旧全局模型写权限
- 最好补充双部门测试夹具，例如：
  - 产品部
  - 产品1部

用于后续验证部门覆盖互不影响。

---

## 三、服务端权限读取与保存

### 7. 抽统一的 scoped 权限读取层

不要把“部门优先、系统兜底”的逻辑散落在页面和 actions 中。

建议统一封装服务端读取方法：

- 获取菜单权限矩阵（按 scope）
- 获取年度指标权限矩阵（按 scope）
- 返回矩阵值的同时，返回哪些格子是 inherited

输入建议：

- `scopeType`
- `departmentOrgNodeId?`

输出建议：

- 当前作用域矩阵
- 每个格子是否为显式值 / 继承值

### 8. 改造菜单权限保存入口

当前入口：

- `saveRoleMenuPermissions()`

需要改成：

- 前端提交 `scopeType`
- 若为部门作用域，再提交 `departmentOrgNodeId`
- 保存系统权限时，只影响系统默认记录
- 保存部门权限时，只影响当前部门覆盖记录

关键要求：

- 不能误删其他部门记录
- 不能误删系统默认记录
- 若支持“取消部门覆盖”，则取消后应删除该部门记录并回退系统默认值

### 9. 改造年度指标权限保存入口

当前入口：

- `saveAnnualGoalRolePermissions()`

改造原则与菜单权限一致：

- 参数改为 scoped
- 仅修改当前作用域记录
- 支持系统默认与部门覆盖的隔离保存

---

## 四、页面结构与交互改造

### 10. 调整组织与权限页的顶部结构

当前结构：

- 顶部 `组织 / 权限` tab
- 部门切换只在 `组织` tab 下出现

目标结构：

1. 顶部先出现 **系统 + 各部门** 的作用域选择器
2. 下方再保留 **组织 / 权限** tab

### 11. 作用域切换行为

#### 选择“部门”时

- `组织` tab：继续展示该部门的小组、成员、主管等
- `权限` tab：展示该部门自己的权限矩阵
- 若某格未显式配置，需要展示为“继承系统”

#### 选择“系统”时

- `权限` tab：展示系统作用域下完整四列权限矩阵
- `组织` tab：显示说明态 / 空态，不复用部门组织内容

### 12. 扩展前端状态模型

前端当前主要围绕：

- `selectedDepartmentOrgNodeId`
- `activeTab`

需要扩成：

- `selectedScopeType`
- `selectedDepartmentOrgNodeId | null`
- `activeTab`

### 13. 为权限矩阵增加“继承态”展示

当前权限格只有“勾选 / 未勾选”两种视觉状态。

需要扩展为至少能区分：

- 显式开启
- 显式关闭
- 继承系统开启
- 继承系统关闭

否则用户无法判断：

- 某格为空是因为部门没配
- 还是因为部门明确禁止

## 五、年度指标权限接入

### 14. 保留权限定义不变

本次不改：

- `annualGoalPermissionDefinitions`
- 权限 code 本身

只改：

- 权限读取来源
- capability 的生成方式

### 15. 把年度指标 capability 改成按 scope 生成

当前年度指标权限能力仍是按全局 `roleType` 读取。

需要改成：

- 给定当前 scope
- 先生成该 scope 下的角色权限矩阵
- 再基于该矩阵生成 `AnnualGoalCapabilities`

### 16. 梳理 ADMIN 的特殊处理

当前 `annual-goal-permissions.ts` 中存在 `ADMIN` 的特殊分支。

这轮需要重点确认：

- `ADMIN` 的数据可见范围是否保留全局：保留
- `ADMIN` 的年度指标 capability 是否仍应从 scoped 权限读取：应接入新作用域模型

也就是说：

- **数据范围不动**
- **权限来源要切到新作用域**

## 六、本轮明确不改的内容

### 17. 不改 `User.roleType`

本轮不推翻现有角色体系：

- `ADMIN`
- `DEPARTMENT_MANAGER`
- `TEAM_LEADER`
- `MEMBER`

用户身份与数据范围仍然由 `User.roleType` 负责。

### 18. 不改 data scope

本轮不改：

- `src/server/permissions/data-scope.ts`

原因：

- 它当前负责数据可见范围
- 本次目标是重构权限配置挂载位置
- 同时改两者会显著扩大回归面

因此：

- `ADMIN` 继续保留全局数据范围
- 部门主管 / 组长 / 成员的数据范围逻辑先保持原样

## 九、重点风险清单

### 1. 假分层风险

如果页面改了，但服务端读取还是全局权限，最终会出现“看起来有系统/部门两层，实际上还是旧逻辑生效”。

### 2. 继承逻辑失真风险

若读取、保存、展示三者不一致，会出现：

- 页面显示继承，实际判定没继承
- 保存一次把继承全固化为脏数据

### 3. SYSTEM 只保留管理员一列的误实现风险

系统作用域必须和部门作用域保持相同矩阵结构，不能缩水成只管理管理员列。

### 4. 年度指标权限绕过新模型风险

如果 `annual-goal-permissions.ts` 里保留旧的全局读取或 `ADMIN` 硬编码特权，年度指标模块会成为新模型的漏口。

### 5. 显式禁止表达能力不足风险

如果继续沿用“有记录=允许”的旧模型，又要求支持“部门显式关闭”，则需要补充额外字段或重新定义记录语义。

## 十、验收与验证清单

### 数据与迁移验证

- 迁移后原全局权限全部进入 `SYSTEM` 作用域
- 没有重复记录
- 老账号登录后系统默认权限不丢失
- 部门未配置时不会复制系统矩阵落库

### 页面交互验证

- 页面顶部出现 `系统 + 各部门` 选择器
- 选择器位于 `组织 / 权限` tab 上方
- 选部门时：组织页显示部门数据，权限页显示部门矩阵
- 选系统时：权限页显示系统矩阵，组织页显示说明态
- 页面能区分继承值与显式值

### 权限隔离与继承验证

- 修改系统权限后，仅影响未覆盖该格子的部门
- 修改系统权限后，不覆盖已有部门显式配置
- 修改某部门权限后，不影响其他部门
- 删除某部门覆盖后，该格回退系统默认值

### 年度指标业务验证

- 使用双部门夹具验证部门隔离
- 验证“年度指标”菜单可见性与按钮权限
- 验证系统默认可兜底，部门覆盖可局部生效
- 验证钉钉同步后组织页与权限页不异常

### 回归边界验证

- `dashboard` 基础访问不受影响
- `KPI` 数据范围不受影响
- `organization` CRUD 不因作用域权限改造而异常
- `data-scope.ts` 行为保持原样
