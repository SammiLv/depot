# 权限作用域重构实施拆分清单

## 目标

这份文档把“权限作用域重构”进一步拆成可直接开工的实施步骤，精确到：

- 先改哪些文件
- 每个文件大概要改什么
- 每一步完成后应验证什么

默认遵循以下约束：

- 不改 `User.roleType`
- 不改 `data-scope.ts` 的数据范围语义
- 先完成“系统默认 + 部门覆盖”的权限挂载模型
- 再接页面和年度指标模块

---

## Phase 1：先立住数据模型

### Step 1.1 修改 Prisma schema

**文件**
- `db/prisma/schema.prisma`

**要做的事**
1. 新增权限作用域枚举，例如：
   - `PermissionScopeType`
   - 值：`SYSTEM`、`DEPARTMENT`
2. 修改 `RoleMenuPermission`
   - 增加 `scopeType`
   - 增加 `departmentOrgNodeId`
   - 调整唯一索引
3. 修改 `RoleAnnualGoalPermission`
   - 增加 `scopeType`
   - 增加 `departmentOrgNodeId`
   - 调整唯一索引
4. 若决定支持“显式关闭”，在这一步同步增加：
   - `allowed Boolean`

**重点决策**
- 这一阶段必须先定：
  - 是否继续沿用“有记录=允许”
  - 还是切到“显式记录 + allowed 布尔值”

**完成后验证**
- Prisma schema 能通过校验
- 新唯一键能表达 SYSTEM 与 DEPARTMENT 两种记录

---

### Step 1.2 生成 migration

**文件**
- `db/prisma/migrations/*`

**要做的事**
1. 生成 schema 变更 migration
2. 编写或补充数据迁移逻辑：
   - 将旧 `RoleMenuPermission` 数据迁到 `SYSTEM`
   - 将旧 `RoleAnnualGoalPermission` 数据迁到 `SYSTEM`
3. 确保历史数据迁完后：
   - 系统默认权限不丢
   - 没有重复记录

**完成后验证**
- 本地迁移可执行
- 迁移后表数据符合预期
- 老权限矩阵可从 `SYSTEM` 作用域读出

---

### Step 1.3 更新 seed

**文件**
- `db/prisma/seed.ts`

**要做的事**
1. 调整菜单权限初始化逻辑
   - 不再按旧全局方式写权限
   - 改为写 `SYSTEM` 作用域默认权限
2. 调整年度指标权限初始化逻辑
   - 改为写 `SYSTEM` 作用域默认权限
3. 如有必要，补充第二个部门夹具
   - 例如“产品1部”

**完成后验证**
- `seed` 可执行
- 系统默认权限能正确初始化
- 至少有两个部门可用于后续覆盖验证

---

## Phase 2：收敛服务端权限读取与保存

### Step 2.1 抽菜单权限 scoped 读取 helper

**建议位置**
- 可放在 `src/server/organization/actions.ts` 附近的私有 helper
- 或抽到新的 organization permission helper 文件

**涉及文件**
- `src/server/organization/actions.ts`
- 如需新文件，可新增 server helper 文件

**要做的事**
1. 给定：
   - `scopeType`
   - `departmentOrgNodeId?`
2. 输出当前权限矩阵
3. 若是部门作用域：
   - 先读部门显式记录
   - 再回退系统默认记录
4. 最好同时返回：
   - 哪些格子是 inherited
   - 哪些格子是 explicit

**完成后验证**
- 同一权限在 SYSTEM 和 DEPARTMENT 下读取结果正确
- 未配置部门记录时能正确回退

---

### Step 2.2 抽年度指标权限 scoped 读取 helper

**涉及文件**
- `src/server/organization/annual-goal-permissions.ts`

**要做的事**
1. 改造当前年度指标权限矩阵获取逻辑
2. 让 capability 来源支持 scope 参数
3. 保持 `annualGoalPermissionDefinitions` 不变
4. 最好让外部调用方不用知道回退细节，只拿最终矩阵和能力

**完成后验证**
- 系统作用域可返回完整四列矩阵
- 部门作用域可正确覆盖系统默认值

---

### Step 2.3 改菜单权限保存入口

**文件**
- `src/server/organization/actions.ts`

**目标函数**
- `saveRoleMenuPermissions()`

**要做的事**
1. 给前端提交参数新增：
   - `scopeType`
   - `departmentOrgNodeId?`
2. 若保存 SYSTEM：
   - 只影响系统默认记录
3. 若保存 DEPARTMENT：
   - 只影响当前部门覆盖记录
4. 若采用“显式覆盖”模型：
   - 支持删除部门覆盖后回退到系统默认

**完成后验证**
- 改系统权限不会误伤任何部门覆盖
- 改某部门权限不会误伤系统或其他部门

---

### Step 2.4 改年度指标权限保存入口

**文件**
- `src/server/organization/actions.ts`

**目标函数**
- `saveAnnualGoalRolePermissions()`

**要做的事**
- 与菜单权限保存逻辑保持一致
- 仅修改当前作用域记录
- 支持回退逻辑

**完成后验证**
- 系统与部门的年度指标权限保存互不干扰

---

### Step 2.5 复用现有部门校验能力

**文件**
- `src/server/organization/actions.ts`
- `src/server/organization/org-tree-utils.ts`

**复用点**
- `findDepartmentOrgNodeId()`
- `assertDepartmentExists()`
- `assertDepartmentManagerScope()`
- `findNearestDepartmentOrgNodeId()`
- `isOrgNodeInSubtree()`

**要做的事**
- 确保部门 scope 的传参与保存都经过合法性校验
- 防止越权修改非当前部门权限

**完成后验证**
- 部门主管只能操作自己作用域内的部门数据

---

## Phase 3：改组织与权限页的数据装配

### Step 3.1 改 `OrgPage` 的数据查询模型

**文件**
- `src/app/(authenticated)/organization/page.tsx`

**要做的事**
1. 保留当前组织数据查询：
   - users
   - orgNodes
   - departments
   - teams
2. 把权限数据查询从“全局矩阵”改成“按当前 scope 获取矩阵”
3. 给页面补充作用域选择器所需数据：
   - `SYSTEM`
   - 当前用户可见的部门列表
4. 明确默认 scope：
   - 管理员可默认进 SYSTEM 或当前部门
   - 部门主管默认进自己所在部门

**完成后验证**
- 页面能拿到 scope selector 的完整数据
- 不同 scope 下的权限矩阵可正确切换

---

### Step 3.2 保持现有组织数据装配尽量不动

**文件**
- `src/app/(authenticated)/organization/page.tsx`

**要做的事**
- 尽量复用当前这些产物：
  - `departments`
  - `teams`
  - `mappedUsers`
  - `teamData`
- 不要因为权限作用域改造而重做组织树映射逻辑

**完成后验证**
- 部门组织视图仍与当前行为一致

---

- `src/app/(authenticated)/organization/content.tsx`

**要做的事**

2. 选择项包括：

3. 将它放在 `组织 / 权限` tab 上方

**完成后验证**
- 页面顶部结构符合“先选作用域，再切 tab”

### Step 4.2 扩前端状态模型

**文件**
- `src/app/(authenticated)/organization/content.tsx`

**要做的事**

3. 确保切换 scope 后：
   - 组织视图

   都同步变化

- 系统 / 部门切换后，页面状态不会串

### Step 4.3 系统作用域下展示空态组织页

**文件**
- `src/app/(authenticated)/organization/content.tsx`

**要做的事**

  - 不复用部门组织内容
  - 显示说明态 / 空态

**完成后验证**
- 用户不会把“系统”误解成一个部门节点

---

- `src/app/(authenticated)/organization/content.tsx`

**要做的事**

- 用户能看明白某格权限来自系统还是部门覆盖

### Step 4.5 改保存 payload

- `src/app/(authenticated)/organization/content.tsx`

1. 提交菜单权限时带上：

3. 若引入 `allowed` 语义，需要同步改前端草稿结构

## 建议开工顺序（最小风险）

### 第一刀
只做：
- schema
- migration
- seed
- scoped permission helper

先不动页面。

### 第二刀
再做：
- `organization/page.tsx`
- `organization/content.tsx`
- scope selector
- 继承态展示

### 第三刀
最后做：
- `annual-goal-permissions.ts`
- 年度指标真实接入
- 全量回归

---

## 关键提醒

这次改造里最容易被低估的，不是 schema，也不是 UI，而是：

1. **权限记录语义是否足够表达显式关闭**
2. **年度指标 capability 是否真的全部切到了新来源**
3. **页面展示的继承状态是否与服务端实际判定完全一致**

如果这三个点没处理干净，就很容易出现“看起来支持系统/部门两层权限，实际上行为还是旧逻辑”的假完成状态。
