# 权限作用域重构字段与接口变更草案

## 目标

这份草案用于提前约定：

- 数据表需要新增哪些字段
- 前后端交互需要新增哪些参数
- 页面加载与权限矩阵展示需要返回什么结构
- 哪些地方需要为“继承态”预留表达能力

这不是最终实现代码，而是为了在真正开改前，把字段设计和接口契约先对齐。

---

## 一、Prisma 数据模型字段草案

### 1. 新增权限作用域枚举

建议新增：

```prisma
enum PermissionScopeType {
  SYSTEM
  DEPARTMENT
}
```

语义：

- `SYSTEM`：系统默认模板
- `DEPARTMENT`：部门级覆盖记录

---

### 2. `RoleMenuPermission` 字段草案

当前大致结构：

```prisma
model RoleMenuPermission {
  id               String   @id @default(cuid())
  roleType         RoleType
  menuPermissionId String
  createdAt        DateTime @default(now())

  @@unique([roleType, menuPermissionId])
}
```

建议改成：

```prisma
model RoleMenuPermission {
  id                  String              @id @default(cuid())
  scopeType           PermissionScopeType
  departmentOrgNodeId String?
  roleType            RoleType
  menuPermissionId    String
  allowed             Boolean             @default(true)
  createdAt           DateTime            @default(now())
  updatedAt           DateTime            @updatedAt

  @@unique([scopeType, departmentOrgNodeId, roleType, menuPermissionId])
  @@index([scopeType, departmentOrgNodeId])
  @@index([menuPermissionId])
}
```

说明：

- 这里把 `allowed` 一并放进草案，是为了支持：
  - 系统默认允许 / 禁止
  - 部门显式允许 / 显式禁止
  - 无部门记录时继承系统
- 若最终决定继续沿用“有记录=允许”，可移除 `allowed`，但届时会失去显式禁止能力

---

### 3. `RoleAnnualGoalPermission` 字段草案

当前大致结构：

```prisma
model RoleAnnualGoalPermission {
  id                     String   @id @default(cuid())
  roleType               RoleType
  annualGoalPermissionId String
  createdAt              DateTime @default(now())

  @@unique([roleType, annualGoalPermissionId])
}
```

建议改成：

```prisma
model RoleAnnualGoalPermission {
  id                     String              @id @default(cuid())
  scopeType              PermissionScopeType
  departmentOrgNodeId    String?
  roleType               RoleType
  annualGoalPermissionId String
  allowed                Boolean             @default(true)
  createdAt              DateTime            @default(now())
  updatedAt              DateTime            @updatedAt

  annualGoalPermission AnnualGoalPermission @relation(fields: [annualGoalPermissionId], references: [id])

  @@unique([scopeType, departmentOrgNodeId, roleType, annualGoalPermissionId])
  @@index([scopeType, departmentOrgNodeId])
  @@index([annualGoalPermissionId])
}
```

---

### 4. 字段语义约定

#### `scopeType`
- 必填
- `SYSTEM` 或 `DEPARTMENT`

#### `departmentOrgNodeId`
- `SYSTEM` 时必须为 `null`
- `DEPARTMENT` 时必须为合法部门 `OrgNode.id`
- 一条记录只对应一个部门，不存多个部门

#### `allowed`
- `true`：该作用域下显式允许
- `false`：该作用域下显式禁止
- 若部门作用域没有该记录，则表示“未显式配置”，应回退系统默认值

---

## 二、服务端内部类型草案

### 1. 作用域输入类型

建议定义：

```ts
type PermissionScopeInput = {
  scopeType: "SYSTEM" | "DEPARTMENT";
  departmentOrgNodeId?: string | null;
};
```

使用场景：

- 读取菜单权限矩阵
- 读取年度指标权限矩阵
- 保存菜单权限
- 保存年度指标权限

---

### 2. 矩阵单元格类型

为了支持“继承态”展示，建议不要只返回 boolean，而是返回结构化单元格：

```ts
type PermissionCellState = {
  allowed: boolean;
  source: "SYSTEM" | "DEPARTMENT";
  inherited: boolean;
  explicit: boolean;
};
```

语义：

- `allowed`：当前最终生效值
- `source`：当前值来自系统还是部门
- `inherited`：是否继承自系统
- `explicit`：是否为当前部门显式配置

对于系统作用域：

- `source = "SYSTEM"`
- `inherited = false`
- `explicit = true`

---

### 3. 矩阵行输出类型

菜单和年度指标都可以共用类似结构：

```ts
type ScopedPermissionMatrixRow = {
  permissionId: string;
  permissionCode: string;
  permissionName: string;
  permissionDescription?: string;
  cells: Record<RoleType, PermissionCellState>;
};
```

---

### 4. 页面作用域选项类型

```ts
type PermissionScopeOption =
  | {
      scopeType: "SYSTEM";
      label: "系统";
      departmentOrgNodeId: null;
    }
  | {
      scopeType: "DEPARTMENT";
      label: string;
      departmentOrgNodeId: string;
    };
```

---

## 三、服务端读取接口草案

这里的“接口”主要指 server-side helper / page data assembler，不一定是 HTTP API。

### 1. 菜单权限矩阵读取

建议 helper 形态：

```ts
async function getScopedMenuPermissionMatrix(
  scope: PermissionScopeInput,
): Promise<ScopedPermissionMatrixRow[]>;
```

输入：

- `scopeType`
- `departmentOrgNodeId?`

输出：

- 当前作用域下的菜单权限矩阵
- 每个格子包含最终值 + 来源 + 是否继承

读取规则：

1. 若 scope 为 `SYSTEM`
   - 直接读系统记录
2. 若 scope 为 `DEPARTMENT`
   - 先读部门显式记录
   - 再读系统默认记录
   - 合并出最终矩阵

---

### 2. 年度指标权限矩阵读取

建议 helper 形态：

```ts
async function getScopedAnnualGoalPermissionMatrix(
  scope: PermissionScopeInput,
): Promise<ScopedPermissionMatrixRow[]>;
```

职责：

- 输出年度指标权限矩阵
- 为 capability 计算提供底层数据

---

### 3. 年度指标 capability 读取

建议 helper 形态：

```ts
async function getScopedAnnualGoalCapabilities(
  roleType: RoleType,
  scope: PermissionScopeInput,
): Promise<AnnualGoalCapabilities>;
```

职责：

- 基于 scoped 权限矩阵生成当前角色能力
- 让上层业务尽量继续依赖 `AnnualGoalCapabilities`
- 避免所有业务层都直接感知“继承 / 覆盖”细节

---

## 四、保存接口草案

当前是 `formData` 驱动的 server action，所以这里重点约定 FormData 字段。

### 1. 菜单权限保存：`saveRoleMenuPermissions()`

建议新增 FormData 字段：

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `scopeType` | `SYSTEM \| DEPARTMENT` | 是 | 当前保存的作用域 |
| `departmentOrgNodeId` | `string` | 部门时必填 | 当前部门 scope 的部门 ID |
| `permissions` | `string` | 是 | JSON 字符串，表示当前矩阵草稿 |

#### 推荐 `permissions` 结构

不要继续只传“已勾选 key 列表”，建议改成更明确的结构：

```ts
type SavePermissionCellInput = {
  roleType: RoleType;
  permissionId: string;
  allowed: boolean;
  explicit: boolean;
};
```

整体提交格式可为：

```ts
type SaveScopedPermissionPayload = {
  scopeType: "SYSTEM" | "DEPARTMENT";
  departmentOrgNodeId?: string | null;
  cells: SavePermissionCellInput[];
};
```

这样能表达：

- 系统下这格是允许还是禁止
- 部门下这格是显式允许、显式禁止，还是取消显式配置回退继承

#### 服务端写入语义建议

##### SYSTEM 保存
- 每个格子都应有明确最终值
- 因为系统层就是默认模板

##### DEPARTMENT 保存
- `explicit = true`：写入部门记录
- `explicit = false`：删除该部门对应记录，表示回退系统默认值

如果保留 `allowed` 字段，则：
- `explicit=true + allowed=true` → 显式允许
- `explicit=true + allowed=false` → 显式禁止
- `explicit=false` → 删除部门记录，回退系统

---

### 2. 年度指标权限保存：`saveAnnualGoalRolePermissions()`

建议与菜单权限保持一致：

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `scopeType` | `SYSTEM \| DEPARTMENT` | 是 | 当前保存的作用域 |
| `departmentOrgNodeId` | `string` | 部门时必填 | 当前部门 scope 的部门 ID |
| `permissions` | `string` | 是 | JSON 字符串，表示当前年度指标权限草稿 |

建议 payload 结构与菜单权限一致，只是 `permissionId` 指向 `AnnualGoalPermission.id`。

## 六、前端草稿状态草案

### 1. 当前问题

现在前端草稿更接近：

- `Set<string>`
- key 形如 `roleType:permissionId`

这种结构只适合“勾选即允许”的全局矩阵，不适合 scoped + inherited 模型。

### 2. 建议草稿结构

```ts
type DraftPermissionCell = {
  roleType: RoleType;
  permissionId: string;
  allowed: boolean;
  explicit: boolean;
};
```

整体草稿：

```ts
type DraftPermissionMatrix = DraftPermissionCell[];
```

或按 key 存：

```ts
type DraftPermissionMap = Record<
  string,
  {
    allowed: boolean;
    explicit: boolean;
  }
>;
```

其中 key 可为：

- `${roleType}:${permissionId}`

## 七、前端交互动作草案

### 1. 权限格点击行为

若支持完整继承语义，建议点击一个格子时不是简单二态切换，而是明确三态或四态切换策略：

#### 部门作用域下
一个格子可能有三种业务状态：

1. 继承系统
2. 显式允许
3. 显式禁止

如果系统默认展示为允许/禁止，则视觉上可扩展成四种最终状态：

- 继承允许
- 继承禁止
- 显式允许
- 显式禁止

因此前端交互需要提前决定：

- 点击是循环切换
- 还是弹出小菜单选择“继承 / 允许 / 禁止”

### 2. 系统作用域下
系统作用域无需“继承”，只有：

- 允许
- 禁止

因此系统层格子可以保持二态切换。

## 十、建议结论

从字段和接口契约角度，这次改造如果想真正支持：

- 系统默认
- 部门覆盖
- 显式允许
- 显式禁止
- 继承展示

那么最稳妥的草案是：

1. **表层引入 `scopeType + departmentOrgNodeId + allowed`**
2. **接口层不要再只传 `Set<string>`，而改为结构化 cell payload**
3. **页面展示层不要再只消费 boolean，而改为消费 cell state**

否则后面越往 UI 和业务判定层走，越容易因为底层表达能力不足而返工。
