# 权限作用域重构工作交接（2026-06-12）

## 1. 今日目标
今天围绕“权限作用域重构”推进了三段工作：
- 把权限模型从“全局 roleType 权限”扩到“SYSTEM / DEPARTMENT 两级作用域”
- 把组织页权限配置改造成按 scope 展示与保存
- 把 annual goal 相关权限判断逐步切到 scoped 权限来源

本次改造目标不是重做用户角色体系，而是在保留 `User.roleType` 的前提下，引入：
- SYSTEM 作为系统默认权限模板
- DEPARTMENT 作为部门覆盖层
- 无部门覆盖记录时，回退继承 SYSTEM

---

## 2. 已完成内容

### 2.1 数据层第一刀：权限表支持作用域
已完成 Prisma 数据模型扩展：
- `RoleMenuPermission`
- `RoleAnnualGoalPermission`

新增/调整的核心字段与语义：
- `scopeType`：`SYSTEM | DEPARTMENT`
- `departmentOrgNodeId`：
  - `SYSTEM` 时约定使用 `""`
  - `DEPARTMENT` 时为对应部门 `OrgNode.id`
- `allowed`：用于表达显式允许/显式关闭，避免旧模型“有记录=允许”无法表达覆盖关闭
- 唯一键已从旧的“role + permission”升级为 scoped 维度

同时：
- 已补 migration
- 已调整 seed，让系统默认权限落在 `SYSTEM` 作用域

备注：今天排查后确认，当前数据库里的 `SYSTEM` 权限记录已经统一用 `departmentOrgNodeId = ""`，没有发现需要额外清洗的 `NULL` 脏数据。

### 2.4 SYSTEM 访问边界已补
已补服务端边界，避免非管理员进入 SYSTEM scope：
- 非 `ADMIN` 不再显示 SYSTEM scope 选项
- 即使手动访问 `?scope=SYSTEM`，也会回退到本部门 scope
- 这已经从“前端不让保存”提升为“服务端不再给非管理员进入 system scope”

## 5. 今天未完成 / 仍需继续的部分

### 5.1 最关键未完成项：真实保存闭环验证
虽然数据层、页面结构、scope 切换和 annual-goal 权限来源已经基本打通，但**今天还没有完成最终的真实写库闭环验证**。

也就是说，以下三类场景还需要实操确认：

1. **SYSTEM 作用域改一格并保存**
   - 验证只影响 `scopeType=SYSTEM, departmentOrgNodeId=""` 的目标记录

2. **DEPARTMENT 作用域改一格并保存**
   - 验证只影响该部门自己的覆盖记录

3. **取消部门显式覆盖后回退继承**
   - 验证删除/清空部门显式配置后，页面与读库都正确回退为 SYSTEM 继承值

这一步非常关键，因为这决定“系统默认 + 部门覆盖 + 回退继承”是否真的闭环。

### 5.2 菜单权限与 annual-goal 权限都要各做一轮
不能只验证菜单权限；还需要对：
- `RoleMenuPermission`
- `RoleAnnualGoalPermission`
分别做一轮保存验证，防止菜单权限正常但 annual-goal 保存语义不一致。

### 5.3 继承态和显式 false 的细节仍需确认
当前已引入 `allowed` 字段来表达显式关闭，但还需要重点确认：
- 前端点击继承态后，是否稳定转换成显式记录
- 提交 payload 里的 `explicit / allowed` 是否与后端 delete/upsert 逻辑完全一致
- 取消覆盖时是否真正删除部门显式记录，而不是留下错误状态
