# 权限作用域重构表结构示例稿

## 一、目标

这份示例稿用于说明：

- `SYSTEM` 作用域记录怎么存
- `DEPARTMENT` 作用域记录怎么存
- 为什么一个 `departmentOrgNodeId` 只对应一个部门
- 为什么多个部门覆盖系统权限时，不是“一条记录存多个部门”，而是“每个部门分别存自己的覆盖记录”

---

## 二、当前旧模型

当前两张权限表本质上都是全局模型：

- `RoleMenuPermission`
- `RoleAnnualGoalPermission`

记录语义大致是：

- 某个角色
- 拥有某个权限

例如：

| roleType | permissionId |
| --- | --- |
| ADMIN | menu_dashboard |
| DEPARTMENT_MANAGER | menu_dashboard |
| TEAM_LEADER | menu_annual_goals |

这里没有“系统默认”或“某部门覆盖”的概念。

---

## 三、改造后的核心语义

改造后，每条记录表达的是：

- 某个作用域
- 某个部门（若为部门作用域）
- 某个角色
- 对某个权限项
- 的一条显式配置

也就是：

- `scopeType`
- `departmentOrgNodeId`
- `roleType`
- `permissionId`

其中：

- `scopeType = SYSTEM` 时，`departmentOrgNodeId = null`
- `scopeType = DEPARTMENT` 时，`departmentOrgNodeId = 某个部门的 OrgNode.id`

---

## 四、SYSTEM 作用域记录示例

系统作用域表示“系统默认模板”。

例如系统默认规定：

- 系统管理员可见“年度指标”菜单
- 部门主管可见“年度指标”菜单
- 组长可见“年度指标”菜单
- 普通成员不可见“年度指标”菜单

若继续沿用“有记录 = 允许”的模型，则系统层可能会存成：

| scopeType | departmentOrgNodeId | roleType | permissionId |
| --- | --- | --- | --- |
| SYSTEM | null | ADMIN | menu_annual_goals |
| SYSTEM | null | DEPARTMENT_MANAGER | menu_annual_goals |
| SYSTEM | null | TEAM_LEADER | menu_annual_goals |

注意：

- 没有 `MEMBER` 那条记录，就意味着普通成员没有该权限
- 这里的 `null` 不是“多个部门”，而是“这是系统默认记录，不归属于任何单一部门”

---

## 五、DEPARTMENT 作用域记录示例

假设：

- 产品部的 `OrgNode.id = dept_product`
- 设计部的 `OrgNode.id = dept_design`

### 示例 1：产品部覆盖系统默认

系统默认：

- `TEAM_LEADER` 可以看年度指标

但产品部想禁掉组长的这个权限。

如果当前模型只支持“有记录 = 允许”，那这里会马上暴露设计问题：

- 没有产品部记录，到底表示“继承系统允许”
- 还是表示“产品部显式禁止”

这说明如果要完整支持“显式关闭”，表结构需要额外表达能力。

### 示例 2：产品部显式允许一个系统默认未开放的权限

假设系统默认：

- 普通成员不能编辑小组指标

但产品部希望：

- 普通成员可以编辑小组指标

则会有一条产品部自己的部门作用域记录：

| scopeType | departmentOrgNodeId | roleType | permissionId |
| --- | --- | --- | --- |
| DEPARTMENT | dept_product | MEMBER | annualGoal.editTeamPlans |

这条记录的意思是：

- 产品部
- 对普通成员
- 显式配置了这个权限

### 示例 3：多个部门各自覆盖

如果设计部也想对普通成员开放同一个权限，则会再有一条记录：

| scopeType | departmentOrgNodeId | roleType | permissionId |
| --- | --- | --- | --- |
| DEPARTMENT | dept_design | MEMBER | annualGoal.editTeamPlans |

所以：

- 不是一条记录里存多个部门
- 而是每个部门分别有自己的记录

---

## 六、为什么 `departmentOrgNodeId` 不能存多个部门

因为这张表的设计目标不是“描述一批部门”，而是“描述一个确定作用域下的一条权限配置”。

如果一条记录里放多个部门，会带来几个问题：

1. 唯一约束无法自然表达
2. 查询某个部门的权限时无法直接走索引
3. 删除某个部门覆盖时会变成数组编辑，而不是删除记录
4. 无法和 Prisma / 关系型数据库常规建模方式一致

因此最自然的关系型建模方式就是：

- 一个部门一条记录
- 多个部门多条记录

---

## 七、推荐唯一键理解方式

推荐把唯一键理解成：

- 在某个作用域里
- 某个部门（若有）
- 某个角色
- 对某个权限项
- 最多只有一条配置记录

即：

- `scopeType + departmentOrgNodeId + roleType + permissionId`

示意：

| scopeType | departmentOrgNodeId | roleType | permissionId | 是否允许重复 |
| --- | --- | --- | --- | --- |
| SYSTEM | null | ADMIN | menu_dashboard | 否 |
| SYSTEM | null | ADMIN | menu_dashboard | 否 |
| DEPARTMENT | dept_product | ADMIN | menu_dashboard | 否 |
| DEPARTMENT | dept_design | ADMIN | menu_dashboard | 否 |

这里：

- 产品部和设计部可以各有一条，因为 `departmentOrgNodeId` 不同
- 同一个部门、同一个角色、同一个权限不能重复

---

## 八、读取逻辑示例

假设当前正在看“产品部”的权限，当前角色是 `TEAM_LEADER`，当前权限是 `annualGoal.viewTeamPlans`。

读取顺序应为：

1. 先查：
   - `scopeType = DEPARTMENT`
   - `departmentOrgNodeId = dept_product`
   - `roleType = TEAM_LEADER`
   - `permissionId = annualGoal.viewTeamPlans`
2. 如果找到了，优先使用部门记录
3. 如果没找到，再查：
   - `scopeType = SYSTEM`
   - `departmentOrgNodeId = null`
   - `roleType = TEAM_LEADER`
   - `permissionId = annualGoal.viewTeamPlans`
4. 若系统也没有，则视为无权限

这就是“部门优先，系统兜底”。

---

## 九、当前模型的关键限制

如果继续沿用“有记录 = 允许，没记录 = 不允许”的旧模型，则它更适合表达：

- 系统默认允许一些权限
- 部门额外增加一些权限

但它不擅长表达：

- 系统默认允许
- 某部门显式禁止

因为“没记录”同时可能表示：

- 继承系统
- 没配
- 显式关闭

这三种语义会冲突。

因此如果你们确认要支持完整的“系统默认 + 部门覆盖（包括显式关闭）”，就应继续评估：

### 方案 A：新增 `allowed` 字段

记录语义变成：

- 这是一条显式配置
- `allowed = true/false`

示例：

| scopeType | departmentOrgNodeId | roleType | permissionId | allowed |
| --- | --- | --- | --- | --- |
| SYSTEM | null | TEAM_LEADER | annualGoal.viewTeamPlans | true |
| DEPARTMENT | dept_product | TEAM_LEADER | annualGoal.viewTeamPlans | false |

这样就能清楚表达：

- 系统默认允许
- 产品部显式关闭

### 方案 B：拆成 allow / deny 两类记录

不太推荐，因为：

- 复杂度更高
- 冲突处理更麻烦
- 可读性更差

从当前项目复杂度看，若要支持显式关闭，`allowed` 布尔值通常是更稳妥的选择。

---

## 十、建议结论

### 若只想支持“系统默认 + 部门额外开放”

那么沿用：

- `scopeType + departmentOrgNodeId + roleType + permissionId`
- 有记录 = 允许

可以成立。

### 若想完整支持“系统默认 + 部门覆盖（可开可关）”

则更推荐：

- `scopeType + departmentOrgNodeId + roleType + permissionId + allowed语义`

否则后续前端展示“继承 / 显式关闭 / 显式开启”时会遇到表达能力不足的问题。

---

## 十一、对当前问题的直接回答

当多个部门都覆盖系统权限时，含义是：

- 每个部门分别保存自己的覆盖记录
- `departmentOrgNodeId` 存的是该部门自己的 `OrgNode.id`
- 不是一个字段里存多个部门
- 也不是一条记录对应多个部门
