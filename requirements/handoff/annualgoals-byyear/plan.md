# 年度指标按年份切换与去归档改造计划

## 目标
将“年度指标”页从“历史记录弹窗 + 归档历史”模式，调整为“按年份切换整个年度指标页面”模式，并移除归档相关能力；同时新增业务约束：同一组织（部门/小组）在同一年仅允许存在一个年度方案。

## 范围
- 页面：`src/app/(authenticated)/annual-goals/content.tsx`
- 查询：`src/server/annual-goals/annual-goals-query.ts`
- 动作：`src/server/annual-goals/actions.ts`
- 如有必要，再同步清理关联权限/复用组件中的归档残留代码

## 约束与原则
1. **先不改数据库结构**：优先通过查询逻辑、UI 结构和服务端校验实现。
2. **若必须改库，先确认**：例如想加唯一索引 `(ownerOrgNodeId, year)` 时，先和你确认。
3. **年份切换作用于整个页面**：不是在弹窗里查看历史详情，而是切换当前页面展示的数据上下文。
4. **历史范围包含当年**：年份切换列表里包含当前年。
5. **唯一规则按“组织 + 年份”生效**：部门和小组分别独立约束。

## 正确需求定义
### 1. 年份切换入口
把当前右上角“历史记录”按钮，改造成“年份切换入口”。

### 2. 年份切换影响范围
当用户切换年份后，以下区域都要跟着切换到该年份的数据：
- 顶部完成度卡片
- 当前方案卡片
- 年度指标 / 元指标 / 季度指标 tab 内容
- 页面内相关统计信息

### 3. 空态规则
即使当前组织在所选年份没有年度方案：
- 页面结构仍保留
- 年度指标 / 元指标 / 季度指标列表仍展示
- 数据为空
- 不再通过“历史记录弹窗”承载年份查看逻辑

## 实施步骤

### 1. 重构 query：从“当前方案 + 历史弹窗数据”改成“按年份驱动页面数据”
把当前查询结果从“activePlan + historyPlansByYear”思路，改成“selectedYear 对应页面视图”思路：
- 收集当前可见范围内所有可用年份
- 为页面提供默认年份（优先当前年，其次最近有方案的年份）
- 按“当前选中组织 + selectedYear”定位页面主方案
- 页面 summary、plan detail、metric/source/quarter 数据都基于 selectedYear 计算

预期结果：
- 页面主内容不再固定绑定“当前生效方案”
- 而是绑定“当前组织在当前选中年份下的方案”

### 2. 改造右上角入口：从“历史记录”改为“年份切换”
建议交互：
- 点击后展示年份列表/下拉菜单
- 可切换 `2026`、`2025`、`2024` 等年份
- 当前选中年份在按钮上或菜单中明确高亮

预期结果：
- 用户理解这是“切换年度视角”而不是“查看归档记录”
- 不再保留旧的历史记录弹窗交互

### 3. 主页面按年份切换数据
改造 `content.tsx`，让以下内容都依赖 `selectedYear`：
- 部门 / 小组切换后的当前方案定位
- 顶部完成度卡片文案与数值
- 方案卡片头部信息
- 年度指标 / 元指标 / 季度指标 tab 内容
- 空态提示文案

预期结果：
- 用户切年份后，看到的是“该组织该年份的完整年度指标页面”
- 而不是仅在弹窗里查看一个历史条目

### 4. 去掉归档功能和相关 UI / server 代码
清理年度指标页中的归档相关元素：
- 去掉卡片右上角“归档”按钮
- 去掉归档确认弹窗
- 去掉恢复按钮与恢复入口
- 去掉“历史记录=归档记录”的提示与文案

同时清理对应代码路径：
- `archiveAnnualGoalPlan`
- `restoreAnnualGoalPlan`
- `archivedPlans`
- 任何依赖 `deletedAt` 解释“历史记录”的前端/服务端逻辑

### 5. 增加“组织 + 年份”唯一校验
在 `createAnnualGoalPlan` 和必要的 `updateAnnualGoalPlan` 流程中增加服务端校验：
- 同一 `ownerOrgNodeId`
- 同一 `year`
- 若已有其他方案存在，则拒绝创建/修改

返回明确错误文案，例如：
- `该组织在 2026 年已存在年度方案，请勿重复创建。`

同时处理编辑场景：
- 如果只是编辑同一条记录本身，不应误判冲突
- 如果允许修改年份或组织归属，则需要排除当前记录 ID 再校验

### 6. 清理残余文案与无用状态
清理这次改造后不再成立的内容：
- “历史记录”命名（如果最终改成“年份切换”）
- “归档后可从历史记录查看和恢复”等文案
- 历史记录弹窗相关 state、detail 组件、交互逻辑
- 任何仅为旧归档模型服务的无用类型、状态、组件和 import

## 验证方案

### 功能验证
1. 默认进入页面时，年份入口正确显示默认年份。
2. 切换年份后，整个页面主内容跟随切换。
3. 当前组织在某年份无方案时，页面结构仍完整，只显示空数据。
4. 页面中不再出现归档按钮、恢复按钮、归档弹窗。
5. 同一组织同一年重复新建方案时，被明确拦截。
6. 不同组织在同一年创建方案，允许。
7. 同一组织不同年份创建方案，允许。

### 代码验证
- 运行：`npm run lint -- "src/app/(authenticated)/annual-goals/content.tsx" "src/server/annual-goals/actions.ts" "src/server/annual-goals/annual-goals-query.ts"`
- 手动在 `http://localhost:3002/annual-goals` 验证：
  - 年份切换入口是否生效
  - 页面主内容是否随年份切换
  - 无方案年份是否正确显示空态
  - 页面无归档入口
  - 创建重复年份方案时报错

## 风险点
1. 当前页面多个区域默认基于“active/current plan”推导，改成“selectedYear 驱动”时，需要统一收口，避免局部仍读旧字段。
2. 若 query 同时返回“当前方案”和“年份方案”，容易造成前端混用，需要尽量简化成单一数据语义。
3. 若未来要从“代码校验”升级为“数据库唯一约束”，需要单独评估历史脏数据与迁移策略。

## 建议落地顺序
1. 先改 query，把页面数据语义切到“selectedYear 驱动”
2. 再改 content，把“历史记录”按钮替换为“年份切换”，并让整页数据随年份变化
3. 再清理剩余历史弹窗 / 归档 UI / 无用状态
4. 最后改 actions，补齐唯一性校验与收尾文案
5. 跑 lint + 页面联调

## 暂不做
- 暂不新增数据库唯一索引
- 暂不重构全部年度指标权限模型
- 暂不把年份切换抽成独立路由页面
