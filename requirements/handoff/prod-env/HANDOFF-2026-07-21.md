## 报障 #1：KPI 模板「删除」按钮无反应

### 现象
- 路径：`/kpi`（KPI 管理）
- "KPI 模板"标签下，每条模板记录右侧的红色「删除」按钮**点了完全没反应**
- 浏览器无任何弹窗、无确认框、无报错（按钮像死的一样）
- "查看 / 编辑 / 启用 / 禁用" 按钮**全部正常**

### 根因
**`src/app/(authenticated)/kpi/content.tsx:128`**

```tsx
{canManageKpiTemplate ? <button type="button" className="text-destructive hover:underline" disabled>删除</button> : null}
```

对比同行的 125–127 行（都正常），第 128 行**同时犯了 3 个错**：

| 错 | 后果 |
|---|---|
| 1. `disabled` 属性 | HTML 原生属性，浏览器对 disabled 按钮**根本不触发 click 事件** |
| 2. 无 `onClick` | 即使去掉 disabled，点了也没函数会执行 |
| 3. 后端无 `deleteKpiTemplate` action | `src/server/kpi/actions.ts` 完全没有删除 KPI 模板的 server action，只有 `deletePersonalKpi`（删个人季度 KPI） |

三件事叠加 = 点了完全没反应。

### 旁证：同文件第 1592–1598 行有一个"正确实现"的删除按钮
季度 KPI 行的删除走完整流程：`onClick → setDeleteQuarterlyKpiRow → Dialog 确认 → deletePersonalKpi → router.refresh()`，模板列表的删除完全没接上这个流程。

### 修复建议（测试环境做）

按"最小 → 更好 → 完整"分级：

#### 最小修复（必须）
1. **`src/server/kpi/actions.ts` 末尾新增**：
   ```ts
   export async function deleteKpiTemplate(templateId: string) {
     // 仿照 actions.ts 里 deletePersonalKpi 的实现模式
     // 软删：update kpiTemplate set deletedAt = new Date() where id = ?
     // 同时 deleteMany kpiTemplateItem where templateId = ?
   }
   ```
2. **`src/app/(authenticated)/kpi/content.tsx:7` import 列表**加上 `deleteKpiTemplate`
3. **第 128 行**改成：
   ```tsx
   {canManageKpiTemplate
     ? <button type="button" className="text-destructive hover:underline" onClick={() => onDelete(row)}>删除</button>
     : null}
   ```
4. **TemplateList 组件 props 加 `onDelete`**，父组件 `KpiContent` 传入完整流程（state → Dialog → 确认 → 调 action → `router.refresh()`），参考第 1251 行 `deleteQuarterlyKpiRow` 的现成写法

#### 更好（推荐）
- 改用项目里 `<Button>` 组件（`@/components/ui-kit`），不要裸用原生 `<button>`
- 删除前用 `Dialog` 二次确认（"确定删除该模板？删除后引用该模板的指标会受影响"）
- `router.refresh()` 后给个 toast 提示

## 报障 #2：设计组添加"小组指标"失败（P2002）

### 现象
- 路径：`/annual-goals`（指标管理） → "小组指标" 标签
- 设计组（`cmrubidnz003zi8tczdy27l61`）用户选某个 sourceMetric（如"创新 ToC 用户增量"）→ 权重 20 → 点「选择」→ **失败**
- 浏览器只看到 Next.js 通用错误："An error occurred in the Server Components render. The specific message is omitted in production builds to avoid leaking sensitive details. A digest property is included on this error instance..."
- 用户感觉"很奇怪"：同页面下产品部能添加、设计组不行

### 根因（生产日志已确认）
**服务日志里真实错误**（生产模式对用户隐藏了，但日志里有完整堆栈）：
```
PrismaClientKnownRequestError P2002:
Invalid `prisma.annualGoalMetric.create()` invocation:
Unique constraint failed on the fields: (`planId`, `sourceMetricId`)
```

**根因不是"设计组"特有**，是结构性 bug：

1. **`db/prisma/schema.prisma:374`**：
   ```prisma
   @@unique([planId, sourceMetricId])
   ```
   这是**物理层 UNIQUE INDEX**，**看不见 `deletedAt` 软删标记**。

2. **`src/server/annual-goals/actions.ts:690-707`** 的 `createAnnualGoalMetric`：
   - 对 TEAM + sourceMetricId 分支**直接 `prisma.annualGoalMetric.create`**
   - **没有 `findFirst` 查重**（包括软删历史）
   - **没有 `try/catch` 包 P2002**
   - **没有"软删豁免"**

3. **设计组 plan 的 metric 历史**（直接查 `db/dev.db` 验证过）：
   ```
   设计组 plan: cmrubidnz003zi8tczdy27l61
     ├─ 单位拓展业绩分值     sourceMetricId=NULL  启用中
     ├─ C 端产品(AGM-004)    sourceMetricId=cmrsqkwfs…  weight=20  已软删(7/21 08:34)
     └─ 创新 ToC 用户增量    sourceMetricId=NULL        weight=20  已软删(7/21 08:48)
   ```
   用户再添加 `C 端产品` 或 `创新 ToC` → `(planId, sourceMetricId)` 物理层重复 → P2002。

### 为什么"产品部没事"——关键对比

| 维度 | 产品部（DEPARTMENT） | 设计组（TEAM） |
|---|---|---|
| 走 `actions.ts` 哪条分支 | 行 740 起的 DEPARTMENT 分支 | 行 676 起的 TEAM + sourceMetricId 分支 |
| 传 `sourceMetricId`？ | **不传**（走 `metricCode` 路径） | **传** |
| NULL + UNIQUE 行为 | SQLite 对 NULL 不视为相等，多行共存不冲突 | `(planId, sourceMetricId)` 都有值，触发 UNIQUE |
| 删除过再添加会怎样？ | 不传 sourceMetricId 就没事 | 软删过 sourceMetricId → 必报 P2002 |

**所以"设计组失败、产品部不失败"是必然**，与设计组权限/配置无关。

### 顺便发现的问题（附带隐患）
- 数据库有 **7 个 ORPHAN plan**：`ownerOrgNodeId` 指向已不存在的 OrgNode（被删过/重建过）
  - 涉及：采购组、采购业务组、B 端组、B 端业务组、C 端组、C 端业务组、设计组（旧版 `cmrsqkwhp…`）
  - 这些 plan 走 `assertPlanEditable` 时会因 `getDescendantOrgNodeIds(不存在的 id)` 返回空而抛"无权维护"
  - 这次报错的 design plan `cmrubidnz…` 是 **7/21 07:14 第二批新建的有效 plan**，不是 ORPHAN 那个
  - **不是本次报障的根因**，但真实隐患，需要清理

### 修复建议（测试环境做）

#### 最小修复（必须）
**`src/server/annual-goals/actions.ts:690` 之前加查重 + 复用：**
```ts
if (sourceMetricId) {
  const sourceMetric = await assertSourceMetricAvailable(sourceMetricId, { ... });

  // 先查是否已存在（含软删）
  const existing = await prisma.annualGoalMetric.findFirst({
    where: { planId, sourceMetricId },
  });

  if (existing) {
    if (existing.deletedAt) {
      // 软删过 → 恢复（清 deletedAt + 改 weight）
      await prisma.annualGoalMetric.update({
        where: { id: existing.id },
        data: { deletedAt: null, weight, updatedBy: user.id },
      });
    } else {
      // 真的重复了
      throw new Error("该指标已添加过，无需重复添加");
    }
  } else {
    await prisma.annualGoalMetric.create({ data: { ... } });
  }
  revalidateAnnualGoals();
  return;
}
```

#### 更好（推荐）
- 把 P2002 在外层 `try/catch` 里识别出来，翻译成中文业务错误（即使最小方案能避免大多数 P2002，也兜底以防漏网）
- `src/app/(authenticated)/annual-goals/content.tsx:356-358` 的 `setError(err.message)` 要识别 `Prisma.PrismaClientKnownRequestError` 的 P2002，给用户友好提示，不要把英文 Prisma 错误原样抛出

#### 配套修复
- **`content.tsx:295-297`** 的 `availableSourceMetrics` 过滤，要考虑"曾经绑定过（含软删）"的 sourceMetric 不能出现在下拉里：
  ```ts
  // 原：plan.metrics 排除了软删 → 用户看到"看似没选过"但实际冲突
  // 新：用 raw 查询或 include deletedAt,过滤时把已绑过(含软删)的全排掉
  ```

#### 可选清理
