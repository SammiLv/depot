# 部门内部管理网站本机开发启动与备用电脑部署方案

## 1. 方案目标

本方案用于指导部门内部管理网站在 MVP 阶段如何本机开发、运行、测试，以及后续如何在没有云服务器的情况下，使用备用电脑进行产品部内网试部署。

当前前提：

```text
1. 暂时没有云服务器。
2. 先在本机跑通 MVP。
3. 后续最多使用一台额外电脑作为试部署机器。
4. 项目首期仅产品部约 15 人试用。
5. 数据库先使用 SQLite。
6. 代码托管在 GitHub。
```

## 2. 总体路线

推荐分三步走：

```text
阶段 1：本机开发和演示
阶段 2：备用电脑内网试部署
阶段 3：后续云端或正式部署
```

对应关系：

```text
本机开发：验证功能是否能跑通
备用电脑：让产品部同事在内网试用
正式部署：试点效果好后再考虑云服务器或部署平台
```

## 5. 本机开发启动方案

### 5.1 本机开发目标

本机阶段主要完成：

```text
1. 初始化项目。
2. 配置 SQLite 数据库。
3. 创建基础数据。
4. 跑通模拟登录。
5. 跑通首页、权限、年度指标、季度工作、KPI 主流程。
```

### 5.2 本机环境准备

需要安装：

```text
Node.js
Git
VS Code 或其他编辑器
```

检查命令：

```bash
node -v
npm -v
git --version
```

### 5.3 本机环境变量

`.env` 示例：

```env
DATABASE_URL="file:./dev.db"
NEXTAUTH_SECRET="local-dev-secret"
APP_URL="http://localhost:3000"
```

MVP 阶段暂不接钉钉时，可以先不配置钉钉参数。

后续接钉钉时再增加：

```env
DINGTALK_APP_KEY=""
DINGTALK_APP_SECRET=""
DINGTALK_AGENT_ID=""
```

### 5.4 本机启动流程

后续项目创建后，本机启动一般为：

```bash
npm install
npx prisma generate
npx prisma migrate dev
npm run seed
npm run dev
```

访问地址：

```text
http://localhost:3000
```

### 5.5 本机开发数据

MVP 阶段建议准备种子数据：

```text
产品部
采购组
B端组
C端组
设计组
管理员用户
部门主管用户
4 个组长用户

年度指标示例
季度工作示例
```

这样可以直接通过模拟登录切换角色测试。

## 6. SQLite 数据库使用方案

### 6.1 本机数据库位置

建议数据库文件位置：

```text
prisma/dev.db
```

环境变量：

```env
DATABASE_URL="file:./dev.db"
```

注意：Prisma 中 SQLite 文件路径是相对于 `prisma/schema.prisma` 所在目录解析，因此 `file:./dev.db` 通常会生成在 `prisma/dev.db`。

### 6.2 是否提交 SQLite 文件到 GitHub

建议：

```text
不提交真实 dev.db 到 GitHub
```

原因：

```text
数据库可能包含真实人员和绩效信息
文件会不断变化
容易造成冲突
不适合版本管理
```

建议提交：

```text
prisma/schema.prisma
prisma/migrations/
prisma/seed.ts
.env.example
```

不提交：

```text
prisma/dev.db
.env
```

`.gitignore` 应包含：

```text
.env
*.db
*.db-journal
prisma/dev.db
```

### 6.3 数据备份

本机或备用电脑使用 SQLite 时，建议定期备份：

```text
每天或每周复制 prisma/dev.db
备份命名：dev-YYYYMMDD.db
保存到单独备份目录
```

例如：

```text
backups/dev-20260602.db
```

如数据开始真实使用，建议不要只依赖百度云盘或同步盘自动同步，最好手动保留多个版本备份。

## 7. GitHub 使用方案

GitHub 用途：

```text
1. 代码托管。
2. 版本管理。
3. 备用电脑拉取代码。
4. 后续可作为自动部署入口。
```

GitHub 不直接承载后端和数据库。

### 7.1 建议提交到 GitHub 的内容

```text
源代码
Prisma schema
数据库迁移文件
种子数据脚本
README
.env.example
```

### 7.2 不建议提交到 GitHub 的内容

```text
.env
SQLite 数据库文件
真实人员数据导出
日志文件
node_modules
构建产物
```

### 7.3 GitHub Pages 说明

GitHub Pages 只适合静态网站。

本项目需要：

```text
登录
数据库
后端 API
审批流
权限判断
定时提醒
```

因此不能只依赖 `github.io` 发布完整系统。

## 8. 备用电脑内网试部署方案

### 8.1 适用场景

当本机 MVP 跑通后，如果需要让产品部同事试用，但暂时没有云服务器，可以使用一台备用电脑作为内网服务器。

适用条件：

```text
备用电脑可以长期保持开机
备用电脑和同事在同一公司网络或可互相访问
公司网络允许访问备用电脑端口
数据无需公网访问
```

### 8.2 备用电脑准备

备用电脑需要安装：

```text
Node.js 20 LTS
Git
浏览器
```

如果是 Windows 电脑，也可以运行，但建议最好使用 macOS、Linux 或 Windows + PowerShell。

### 8.3 从 GitHub 拉代码

备用电脑上：

```bash
git clone <项目 GitHub 地址>
cd department-management
npm install
```

### 8.4 配置环境变量

在备用电脑创建 `.env`：

```env
DATABASE_URL="file:./prod.db"
NEXTAUTH_SECRET="请换成一串较长随机字符"
APP_URL="http://备用电脑IP:3000"
```

说明：

```text
备用电脑建议使用 prod.db，避免和本机开发数据库混淆。
```

### 8.5 初始化数据库

```bash
npx prisma generate
npx prisma migrate deploy
npm run seed
```

如果只是试用，也可以使用：

```bash
npx prisma migrate dev
npm run seed
```

但正式一点建议用 `migrate deploy`。

### 8.6 构建和启动

```bash
npm run build
npm run start
```

默认访问：

```text
http://localhost:3000
```

其他同事访问：

```text
http://备用电脑IP:3000
```

示例：

```text
http://192.168.1.20:3000
```

### 8.7 查看备用电脑 IP

macOS：

```bash
ipconfig getifaddr en0
```

Windows：

```bash
ipconfig
```

查看 IPv4 地址。

### 8.8 防火墙设置

如果同事打不开，需要检查：

```text
备用电脑防火墙是否允许 3000 端口
公司网络是否允许局域网访问
备用电脑和访问者是否在同一网段
```

可考虑换成 80 或 8080 端口，但 MVP 阶段 3000 即可。

## 9. 备用电脑运行稳定性建议

### 9.1 保持服务运行

简单方式：

```bash
npm run start
```

缺点：关闭终端后服务停止。

更稳定方式可后续使用：

```text
PM2
```

PM2 可以让 Node 服务后台运行，并在异常退出后自动重启。

后续命令示例：

```bash
npm install -g pm2
pm2 start npm --name department-management -- run start
pm2 save
```

MVP 初期可以先不用 PM2，等需要多人试用时再加。

### 9.2 数据备份

备用电脑上建议定期备份：

```text
prisma/prod.db
```

备份频率：

```text
试用期：每天一次
稳定后：每周一次
重要调整前：手动备份一次
```

备份目录建议：

```text
backups/
├── prod-20260602.db
├── prod-20260603.db
└── prod-20260604.db
```

### 9.3 更新代码流程

后续代码更新时：

```bash
git pull
npm install
npx prisma migrate deploy
npm run build
pm2 restart department-management
```

如果不用 PM2：

```bash
git pull
npm install
npx prisma migrate deploy
npm run build
npm run start
```

## 10. 本机开发与备用电脑数据关系

建议区分两套数据库：

```text
本机：dev.db，用于开发和测试
备用电脑：prod.db，用于产品部试用
```

不要频繁把本机数据库覆盖备用电脑数据库。

如果需要初始化试用数据，建议通过种子脚本或导入脚本完成。

## 11. 后续迁移云端方案

当备用电脑试用稳定后，如果需要正式推广，可以迁移到：

```text
云服务器
Render
Railway
Vercel + 外部数据库
```

正式部署建议改为：

```text
PostgreSQL
```

迁移方向：

```text
SQLite prod.db
→ 导出数据
→ PostgreSQL
→ 修改 DATABASE_URL
→ 部署应用
```

## 12. 安全注意事项

MVP 本机阶段：

```text
可以使用模拟登录
不要录入过多敏感真实数据
```

备用电脑试用阶段：

```text
建议只在公司内网访问
不要直接暴露公网
不要把 .env 上传 GitHub
不要把 SQLite 数据库上传 GitHub
定期备份数据库
至少保留管理员账号
```

后续正式阶段：

```text
接入钉钉登录
启用 HTTPS
使用强随机密钥
完善权限校验
完善备份机制
记录关键操作日志
```

## 13. 推荐实施顺序

### 13.1 本机阶段

```text
1. 初始化 Next.js 项目。
2. 配置 Tailwind CSS 和 shadcn/ui。
3. 配置 Prisma 和 SQLite。
4. 创建数据库 schema。
5. 创建种子数据。
6. 实现模拟登录：居中账号确认页、头像/姓名、立即登录按钮、用户选择返回态。
7. 实现基础布局和导航。
8. 实现首页工作台。
9. 实现年度指标、季度工作、KPI 主流程。
10. 本机完整测试。
```

### 13.2 备用电脑阶段

```text
1. 创建 GitHub 仓库。
2. 本机推送代码到 GitHub。
3. 备用电脑安装 Node.js 和 Git。
4. 备用电脑拉取代码。
5. 配置 .env。
6. 初始化 SQLite 数据库。
7. 构建并启动应用。
8. 同事通过内网 IP 访问。
9. 收集反馈。
10. 迭代优化。
```

### 13.3 后续正式阶段

```text
1. 评估是否需要云服务器。
2. 评估是否迁移 PostgreSQL。
3. 接入钉钉登录和消息提醒。
4. 配置正式域名和 HTTPS。
5. 建立备份机制。
6. 再考虑服务拆解。
```

项目创建后常用命令：

```bash

```

具体命令以后以项目实际 `package.json` 为准。

## 15. 设计结论

当前阶段最合理的部署路线是：

```text
先本机跑通 MVP
再用备用电脑做内网试部署
GitHub 负责代码托管和版本发布
SQLite 负责低成本数据存储
后续根据实际使用效果再迁移云端和 PostgreSQL
```

这条路线成本最低、实现最快、维护复杂度最低，也符合产品部小范围试点的当前条件。
