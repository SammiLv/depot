# 部门内部管理网站技术与部署建议方案

## 1. 推荐结论

本项目建议采用“轻量化全栈 + 清晰分层 + SQLite 起步 + 后续可迁移”的路线。

推荐方案：

```text
Next.js 单仓全栈项目
+ TypeScript
+ Prisma
+ SQLite 本机数据库
+ 后续可迁移 PostgreSQL
+ GitHub 托管代码并作为发布入口
+ 本机先跑通 MVP
+ 备用电脑做内网试部署
+ 后续如有需要再迁移云服务器或拆分服务
```

不建议一开始采用传统复杂架构：

```text
Vue 前端独立项目
+ NestJS 后端独立项目
+ PostgreSQL
+ 微服务拆分
```

原因是当前首期仅产品部约 15 人试用，未来即使推广至公司内部也不到 150 人，业务规模和并发压力不大。项目当前更需要快速验证、方便维护、降低部署和运维复杂度。

## 2. 架构原则

采用“代码仓库一体化，但职责清晰分层”的方式。

也就是说，不拆成两个独立项目，但在代码内部明确区分：

```text
- 前端页面层
- 后端 API 层
- 业务服务层
- 数据库访问层
- 权限控制层
- 审批流层
- 通知提醒层
```

这样可以兼顾：

- 本机开发简单
- 部署简单
- 后续维护成本低
- 前后端职责不混乱
- 未来如果需要拆服务，也有清晰边界

## 3. 推荐技术栈

### 3.1 应用框架

推荐使用：

```text
Next.js + TypeScript
```

原因：

- 一个项目内同时支持页面和后端 API。
- 不需要维护前端、后端两个独立工程。
- 适合中小型内部管理系统。
- 后续可以通过 GitHub 进行版本管理和自动部署。
- 与 Prisma、SQLite、PostgreSQL 配合成熟。

### 3.2 UI 组件

可选：

```text
Ant Design
或
shadcn/ui
```

如果优先后台管理系统效率，建议先用 Ant Design。

原因：

- 表格、表单、弹窗、审批列表等后台组件完整。
- 对内部管理系统更直接。
- 开发速度快。

### 3.3 数据库 ORM

推荐：

```text
Prisma
```

原因：

- 可以用统一方式操作 SQLite 和 PostgreSQL。
- 数据模型清晰。
- 方便后续迁移数据库。
- 避免在代码中写大量原生 SQL。

### 3.4 数据库

本机和试点阶段：

```text
SQLite
```

正式推广阶段：

```text
PostgreSQL
```

SQLite 对 15 人试点完全够用，对 150 人以内的低频内部管理场景也可以支撑。但如果后续需要更稳定的多人并发写入、备份、权限和长期运维，建议迁移到 PostgreSQL。

## 5. 代码分层约束

为了避免后续业务扩展导致耦合度变高，建议从一开始遵守以下规则：

```text
1. 页面只负责展示和用户交互。
2. 页面不直接访问数据库。
3. API 只负责接收请求、校验权限、调用业务服务、返回结果。
4. 业务规则统一放在 server/services。
5. 数据库操作统一放在 server/repositories。
6. 权限判断统一放在 server/permissions。
7. 审批流程统一放在 server/workflows。
8. 钉钉提醒统一放在 server/notifications。
```

这样即使后续新增培训管理、项目复盘、会议管理、OKR 管理等模块，也可以按模块扩展，不会把原有代码搅乱。

## 6. 为什么不建议一开始前后端完全分离

传统前后端分离通常是：

```text
frontend/
backend/
```

这种方式虽然边界清晰，但会增加维护成本：

- 两套启动命令。
- 两套项目配置。
- 两套路由和接口配置。
- 跨域配置。
- 登录态传递。
- API 地址环境变量。
- 前后端联调成本。
- 两套部署流程。

对当前 15 人试点项目来说，这些复杂度暂时没有必要。

更合理的是：

```text
单仓全栈，内部严格分层
```

后续如果业务复杂度真的上升，可以从 `src/server` 层逐步拆出独立后端服务。

## 7. SQLite 使用建议

### 7.1 当前阶段是否够用

够用。

该项目主要数据包括：

- 用户和组织数据。
- 年度指标。
- 季度工作计划。
- 月度计划。
- 周更新。
- KPI 模板和评分。
- 人才画像。
- 奖励台账。
- 审批记录。
- 预警记录。

这些数据量较小，15 人试点完全没有压力。

### 7.2 使用 SQLite 的注意事项

- 数据库文件需要定期备份。
- 不适合大量用户同时频繁写入。
- 不建议部署到不支持持久化文件的平台。
- 不要在代码里大量写原生 SQLite SQL。
- 尽量通过 Prisma 操作数据库，方便后续迁移。

### 7.3 后续迁移 PostgreSQL

如果使用 Prisma，后续迁移相对方便。

本机 SQLite：

```env
DATABASE_URL="file:./dev.db"
```

后续 PostgreSQL：

```env
DATABASE_URL="postgresql://user:password@host:5432/dbname"
```

迁移步骤大致为：

```text
1. 修改 Prisma datasource provider。
2. 调整少量 SQLite 和 PostgreSQL 字段差异。
3. 生成 PostgreSQL 数据库迁移。
4. 导出 SQLite 数据。
5. 导入 PostgreSQL。
6. 修改部署环境变量。
7. 验证业务功能。
```

## 8. GitHub 发布方式说明

GitHub 可以承担两个角色：

```text
1. 代码仓库：保存和管理项目代码。
2. 发布入口：通过 GitHub Actions 或部署平台自动拉取代码发布。
```

但 GitHub Pages 不适合作为本项目的最终运行环境。

原因是本项目需要：

- 登录。
- 后端 API。
- 数据库。
- 审批流。
- 定时任务。
- 钉钉集成。
- 服务端权限判断。

GitHub Pages 只能托管静态前端页面，不能完整承载这些能力。

## 9. 推荐部署路线

### 9.1 阶段 1：本机跑通 MVP

运行环境：

```text
你的电脑
Next.js
SQLite
Prisma
```

访问方式：

```text
http://localhost:3000
```

目标：

- 跑通页面。
- 跑通基础数据。
- 跑通权限。
- 跑通指标、季度工作、KPI 主流程。

### 9.2 阶段 2：备用电脑内网试部署

如果暂时没有云服务器，可以使用额外一台电脑作为小型内网服务器。

运行环境：

```text
备用电脑
GitHub 拉取代码
Next.js 生产运行
SQLite 数据库文件保存在本机磁盘
```

同事访问方式：

```text
http://备用电脑内网IP:3000
```

示例：

```text
http://192.168.1.20:3000
```

适用场景：

- 产品部 15 人试用。
- 公司内网访问。
- 暂时不购买云服务器。

注意事项：

- 备用电脑需要保持开机。
- 网络需要稳定。
- 公司内网需要允许访问对应端口。
- SQLite 数据库文件需要定期备份。
- 不建议直接暴露公网。
- 如需公网访问，应先完善登录、安全、HTTPS 和备份机制。

### 9.3 阶段 3：正式部署

如果试点效果好，再升级为：

```text
云服务器 / Render / Railway / Vercel
+ PostgreSQL
+ 正式域名
+ HTTPS
+ 数据库备份
```

此阶段再考虑：

- PostgreSQL 数据库。
- 自动部署。
- 日志监控。
- 数据备份。
- 权限审计。
- 钉钉正式集成。

## 10. 后续服务拆解策略

当前不建议拆服务。

但可以提前保留拆分边界：

```text
src/server/services       业务逻辑
src/server/repositories   数据库访问
src/server/permissions    权限逻辑
src/server/workflows      审批流
src/server/notifications  通知提醒
```

如果未来业务规模扩大，可以优先拆：

```text
1. 通知提醒服务
2. 定时任务服务
3. 审批流服务
4. 独立后端 API 服务
```

不需要一开始全部拆分。

## 11. 推荐实施顺序

建议接下来按以下顺序推进：

```text
1. 确认需求规格说明书。
2. 输出 MVP 功能清单。
3. 输出页面清单和页面流程。
4. 输出数据库表设计。
5. 搭建 Next.js + Prisma + SQLite 项目骨架。
6. 先实现模拟登录和基础权限，登录页采用居中账号确认卡片和 RJ 蓝色主按钮。
7. 实现年度指标管理。
8. 实现季度工作管理。
9. 实现 KPI 管理。
10. 做本机测试。
11. 再部署到备用电脑供同事内网访问。
```

## 12. 最终建议

当前最适合的方案不是企业级复杂架构，而是：

```text
轻量启动
清晰分层
先跑通 MVP
用 SQLite 降低本地开发和试点成本
用 Prisma 保留迁移能力
用 GitHub 管理代码和发布流程
用备用电脑完成内网试部署
后续根据实际使用情况再升级数据库、部署环境或拆分服务
```

这条路线更适合当前产品部试点，也更符合后续低成本维护和逐步扩展的需求。
