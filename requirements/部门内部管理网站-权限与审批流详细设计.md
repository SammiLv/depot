# 部门内部管理网站权限与审批流详细设计

## 1. 设计目标

权限与审批流是本系统的核心基础能力，MVP 阶段需要先满足产品部内部管理试点，重点解决：

```text
谁能登录系统
谁能看哪些菜单
谁能看哪些数据
谁能提交事项
谁能审批事项
审批通过后业务状态如何变化
驳回后如何修改再提交
```

设计原则：

```text
1. 先简单可用，不做过度复杂权限系统。
2. 菜单权限和数据权限分开设计。
3. 数据权限按组织层级控制。
4. 审批流统一抽象，避免每个模块重复写审批逻辑。
5. MVP 阶段先支持产品部单部门，后续可扩展多部门。
```

## 2. 角色定义

MVP 阶段采用 4 类角色。

| 角色 | 角色编码 | 说明 |
|---|---|---|
| 初始管理员 | ADMIN | 系统初始化、基础配置、全量数据权限 |
| 部门主管 | DEPARTMENT_MANAGER | 管理产品部全量业务数据和审批 |
| 组长 | TEAM_LEADER | 管理本组成员数据和审批 |
| 普通成员 | MEMBER | 维护本人业务数据 |

## 3. 权限模型

系统权限分为两类：

```text
菜单权限
数据权限
```

### 3.1 菜单权限

菜单权限控制用户能进入哪些功能页面。

示例：

```text
首页工作台
年度指标
季度工作
KPI 管理
人才发展
组织管理
权限设置
通知中心
```

菜单权限通过以下表维护：

```text
MenuPermission
RoleMenuPermission
```

MVP 阶段采用“角色 → 菜单”的静态配置即可，不需要复杂按钮级权限。

### 3.2 数据权限

数据权限控制用户能看到哪些业务数据。

规则：

| 角色 | 数据范围 |
|---|---|
| ADMIN | 全部数据 |
| DEPARTMENT_MANAGER | 产品部全部数据 |
| TEAM_LEADER | 本组数据 |
| MEMBER | 本人数据 |

数据权限判断依据：

```text
user.id
user.teamId
user.departmentId
user.roleType
```

## 7. 审批流总体设计

MVP 阶段统一审批流适用于：

```text
年度指标方案和季度目标拆解
季度工作规划
个人 KPI 制定
```

统一审批表：

```text
ApprovalInstance
ApprovalTask
```

业务表自身也保存审批状态，方便列表查询和页面展示。

审批实例负责记录：

```text
审批对象
提交人
当前审批人
审批状态
提交时间
完成时间
```

审批任务负责记录：

```text
审批节点
审批人
处理状态
审批意见
处理时间
```

## 8. 通用审批状态

```text
DRAFT              草稿
PENDING_LEADER     待组长审批
PENDING_MANAGER    待主管审批
APPROVED           已通过
REJECTED           已驳回
WITHDRAWN          已撤回
```

状态流转：

```text
草稿
  ↓ 提交
待组长审批
  ↓ 组长通过
待主管审批
  ↓ 主管通过
已通过
```

驳回流转：

```text
待组长审批 / 待主管审批
  ↓ 驳回
已驳回
  ↓ 提交人修改
草稿
  ↓ 重新提交
待组长审批
```

撤回流转：

```text
待组长审批 / 待主管审批
  ↓ 提交人撤回
已撤回
  ↓ 修改
草稿
```

## 9. 年度指标审批流

### 9.1 场景

年度指标审批主要针对小组年度指标方案和小组季度目标拆解结果。

流程：

```text
主管创建部门年度指标方案
→ 拆分/下发小组年度指标方案
→ 组长维护年度指标项、权重和季度目标
→ 提交
→ 组长审批
→ 主管审批
→ 审批通过
→ 作为季度 KPI 候选参考项
```

### 9.2 状态流转

```text
DRAFT
→ PENDING_LEADER
→ PENDING_MANAGER
→ APPROVED
```

驳回：

```text
PENDING_LEADER → REJECTED
PENDING_MANAGER → REJECTED
REJECTED → DRAFT
```

### 9.3 审批人规则

| 节点 | 审批人 |
|---|---|
| 组长审批 | 提交人所属小组组长 |
| 主管审批 | 提交人所属部门主管 |

如果提交人本人是组长：

```text
跳过组长审批，直接进入主管审批
```

如果提交人是部门主管：

```text
可由管理员审批，或 MVP 阶段直接通过
```

MVP 建议：部门主管本人的指标由管理员审批，若无管理员参与，可先直接通过。

### 9.4 通过后的动作

审批通过后：

```text
1. AnnualGoalPlan.approvalStatus 更新为 APPROVED。
2. AnnualGoalPlan.approvedAt 写入当前时间。
3. AnnualGoalMetric 和 AnnualGoalQuarterTarget 进入可更新状态。
4. 季度目标可作为 PersonalKpiItem 候选参考项。
5. 创建通知给提交人。
6. 首页待办数量更新。
```

### 9.5 年中调整版本审批

年中调整复用年度指标审批流。

```text
复制已生效 AnnualGoalPlan
→ 生成新 version
→ 调整 AnnualGoalMetric / AnnualGoalQuarterTarget
→ 填写 revisionReason
→ 提交审批
→ 审批通过后新版本生效
→ 旧版本 isActive = false
→ 写入 AnnualGoalRevisionLog
```

## 10. 季度工作审批流

### 10.1 场景

成员每季度末制定下季度工作规划，提交审批。

流程：

```text
成员创建季度工作计划
→ 提交
→ 组长审批
→ 主管审批
→ 审批通过
→ 进入执行和周更新阶段
```

### 10.2 状态流转

```text
DRAFT
→ PENDING_LEADER
→ PENDING_MANAGER
→ APPROVED
```

组长本人提交：

```text
DRAFT
→ PENDING_MANAGER
→ APPROVED
```

### 10.3 通过后的动作

```text
1. QuarterlyWork.approvalStatus 更新为 APPROVED。
2. QuarterlyWork.status 初始化为 NOT_STARTED 或 IN_PROGRESS。
3. 创建后续月度计划入口。
4. 通知提交人审批通过。
```

### 10.4 驳回后的动作

```text
1. QuarterlyWork.approvalStatus 更新为 REJECTED。
2. 记录审批意见。
3. 通知提交人修改。
4. 提交人修改后重新提交。
```

## 11. KPI 审批与评分流

KPI 流程比普通审批更长，分为两个阶段：

```text
KPI 制定审批阶段
KPI 评分阶段
```

### 11.1 KPI 制定审批阶段

普通成员：

```text
草稿
→ 提交 KPI
→ 组长审批
→ 主管审批
→ 待自评
```

组长：

```text
草稿
→ 提交 KPI
→ 主管审批
→ 待自评
```

状态：

```text
DRAFT
PENDING_LEADER
PENDING_MANAGER
PENDING_SELF_REVIEW
```

### 11.2 KPI 评分阶段

普通成员：

```text
待自评
→ 成员自评
→ 待组长评分
→ 组长评分
→ 待主管终评
→ 主管终评
→ 已完成
```

组长：

```text
待自评
→ 组长自评
→ 待主管终评
→ 主管终评
→ 已完成
```

状态：

```text
PENDING_SELF_REVIEW
PENDING_LEADER_SCORE
PENDING_MANAGER_SCORE
COMPLETED
```

### 11.3 KPI 驳回规则

审批阶段可驳回：

```text
PENDING_LEADER → REJECTED
PENDING_MANAGER → REJECTED
REJECTED → DRAFT
```

评分阶段一般不建议驳回，若需要调整，由管理员或主管退回到对应状态。

MVP 阶段建议简化：

```text
评分阶段不做驳回，只允许保存评分和修改评分，直到主管终评完成。
```

### 11.4 KPI 终评后动作

主管终评完成后：

```text
1. PersonalKpi.status 更新为 COMPLETED。
2. 写入 finalScore。
3. 更新所有 PersonalKpiItem.finalScore。
4. 生成 KPI 结果通知。
5. 人才画像中可展示最近一年 KPI 结果。
```

## 16. MVP 简化策略

为了降低首期复杂度，MVP 阶段建议简化：

```text
1. 一个用户只有一个主角色。
2. 不做按钮级权限配置，只在代码中控制关键操作。
3. 不做复杂多部门，只支持产品部。
4. 组长由 Team.leaderId 指定。
5. 主管由 Department.managerId 指定。
6. 主管本人的审批可由管理员处理，或 MVP 阶段直接通过。
7. 通知只做系统内通知，不接钉钉。
8. 周更新提醒先动态查询，不做定时任务。
```

## 17. 权限与审批验收标准

```text
成员登录后只能看本人指标、本人季度工作、本人 KPI、本人画像。
组长登录后能看本组数据，不能看其他组数据。
主管登录后能看产品部全部数据。
管理员登录后能看全部数据和权限配置页面。
成员不能访问组织管理和 KPI 模板管理页面。
组长不能访问角色权限配置页面。
```

### 17.2 审批验收

```text
成员提交指标后，组长能看到审批待办。
组长通过后，主管能看到审批待办。
主管通过后，指标状态变为已通过。
组长驳回后，成员能修改并重新提交。
成员提交季度工作后，能按相同流程审批。
成员提交 KPI 后，能完成审批、自评、组长评分、主管终评。
组长本人的 KPI 跳过组长审批和组长评分。
```

### 17.3 待办通知验收

```text
审批提交后生成审批人待办。
审批处理后关闭当前待办。
审批通过或驳回后通知提交人。
KPI 到自评阶段后生成本人待办。
KPI 到评分阶段后生成组长或主管待办。
```

## 18. 设计结论

本权限与审批流设计采用“角色简单、数据范围清晰、审批统一抽象”的思路：

```text
菜单权限控制入口
数据权限控制可见范围
操作权限控制能否执行动作
通用审批表记录流程
业务表保留当前状态
待办和通知提升使用体验
```

该设计能支持 MVP 快速落地，也能为后续接入钉钉、扩展多部门、增加更多审批事项保留空间。
